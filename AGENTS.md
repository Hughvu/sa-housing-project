# AGENTS.md

## Scope

These instructions apply to the entire repository. More specific `AGENTS.md`
files may be added in subdirectories later; if present, the closest applicable
file takes precedence for that directory.

## Project goal

This is a Python and Streamlit Business Analyst portfolio project for South
Australian housing analytics.

The project should provide a transparent, reproducible decision-support view of:

- rental-cost pressure;
- population growth;
- dwelling approvals and housing-supply pipeline indicators; and
- data quality and analytical limitations.

The dashboard is not a forecast, a measure of individual household rental
stress, or an infrastructure-capacity model. Do not describe it as one unless
the methodology and supporting data are materially expanded and validated.

Analytical correctness, traceability and honest limitations take priority over
visual polish.

## Folder structure

- `app.py` — Streamlit presentation and interaction layer.
- `src/` — reusable production code.
  - `pipeline.py` — raw-to-processed transformations and validation.
  - `scoring_model.py` — pressure-index calculations and scoring rules.
- `tests/` — automated tests for production behaviour and data integrity.
- `data/raw/` — immutable copies of authoritative source files.
- `data/processed/` — reproducible outputs generated by the pipeline.
- `docs/` — data dictionary, source register and supporting documentation.
- `notebooks/` — exploration only; notebooks are not the production pipeline.
- `assets/` — dashboard screenshots and presentation artifacts.
- `.github/workflows/` — continuous-integration configuration.
- `README.md` — user-facing setup, methodology and project overview.
- `project_status.md` — chronological record of material changes, validation
  results, source updates and remaining risks.

Do not introduce a second implementation of pipeline or scoring logic inside
`app.py` or a notebook. Production logic belongs under `src/`.

## Data handling

### Source quality

- Prefer authoritative government sources such as the Australian Bureau of
  Statistics, SA Housing Trust, AIHW and PlanSA.
- Record every source in `docs/source_register.md`, including authority,
  reference period, local filename, URL, use and limitations.
- Never fabricate, interpolate or silently replace unavailable values.
- Clearly distinguish observed data, calculated fields, assumptions and
  analytical proxies.

### Raw data

- Treat `data/raw/` files as immutable.
- Do not overwrite an earlier release with a newer source file.
- Preserve source filenames when practical.
- Do not modify Excel workbooks, ZIP archives or other raw artifacts in place.
- Do not commit credentials, API keys, personal information or restricted data.

### Processed data

- Generate processed files through `python -m src.pipeline`.
- Do not hand-edit generated CSV files.
- Keep transformations deterministic and auditable.
- Validate schemas before selecting columns by name or position.
- Preserve numeric and date types; do not store formatted display strings when
  a typed value is appropriate.
- Fail clearly on missing files, worksheets, columns, duplicate keys, invalid
  joins or impossible values.
- Document any output-schema change in `docs/data_dictionary.md` and
  `project_status.md`.

### Analytical safeguards

- Keep all analysis at a consistent geography and period unless a difference is
  explicitly disclosed.
- Never compare a partial year with a complete year without a visible YTD label
  or like-for-like comparison.
- Building approvals are permits, not commencements or completed dwellings.
- Rental-bond data does not represent all tenancies, asking rents or vacancy.
- Counts of 1–5 in the rent source are suppressed; respect the minimum-sample
  eligibility rule.
- The current rent-to-income measure uses 2021 Census income and must always be
  described as a screening proxy.
- The Housing Pressure Index is relative to eligible South Australian areas. It
  is not an absolute shortage or financial-stress measure.
- Changes to weights, thresholds, eligibility rules, ranking direction or tie
  handling require tests and documentation.

## Coding style

- Support Python 3.10 or newer unless project configuration states otherwise.
- Follow PEP 8 and use four-space indentation.
- Use type hints for public functions and non-obvious data structures.
- Write concise docstrings explaining purpose, inputs and important assumptions.
- Prefer small, single-purpose functions over long procedural blocks.
- Use `pathlib.Path` for filesystem paths.
- Keep constants at module level and use descriptive names.
- Avoid unexplained magic numbers.
- Avoid broad exception handling. Catch only errors that can be handled or
  translated into a clearer message.
- Do not use mutable default arguments.
- Preserve deterministic ordering in generated outputs.
- Avoid unnecessary dependencies; add a runtime package only when production
  code directly needs it.
- Keep formatting-only changes separate from behavioural changes when possible.
- Do not refactor unrelated code during a targeted task.

## Streamlit app rules

- `app.py` is the presentation layer; it may load processed data but must not
  reproduce pipeline or scoring calculations.
- Use cached loading for stable local datasets.
- Validate required columns before rendering charts.
- Handle missing files, empty filters and no-result states gracefully.
- Keep category labels, colours and number formats consistent across tabs.
- Use accessible colour contrast and do not rely on colour alone to communicate
  meaning.
- Ensure charts have clear titles, axis labels, units and tooltip context.
- Identify partial periods, original or adjusted series, proxy measures and
  sample-quality limitations next to the relevant output.
- Do not imply causation from descriptive comparisons.
- Keep downloads auditable and use descriptive filenames.
- Do not add an infrastructure-readiness score without validated capacity data
  and documented methodology.
- Test every tab at desktop and narrow browser widths after material UI changes.
- Avoid deprecated Streamlit APIs and review runtime warnings.

## Testing rules

- Run the pipeline and test suite before handing off production changes:

  ```powershell
  python -m src.pipeline
  pytest -q
  ```

- Add or update tests for every bug fix or behavioural change.
- Test boundary values, missing data, suppressed samples, duplicate keys,
  unmatched joins, zero denominators, ties and invalid weights.
- Test YTD behaviour independently of a hard-coded current month.
- Assert output schemas and key invariants rather than only row counts.
- Keep tests deterministic and independent of network access.
- Tests must use committed fixtures or raw source files; they must not alter raw
  data.
- Do not weaken, delete or skip a failing test solely to make CI pass.
- If a production defect is discovered outside an agent's allowed files, report
  it instead of editing outside scope.
- GitHub Actions must remain compatible with Linux and the supported Python
  version.
- Browser or Streamlit smoke testing supplements unit tests; it does not replace
  them.

## Git safety rules

- Inspect `git status` before starting and before handing off work.
- Treat pre-existing changes as user-owned.
- Do not overwrite, revert, stage or delete unrelated modifications.
- Do not use `git reset --hard`, destructive checkout commands, force pushes or
  history rewriting unless the user explicitly requests them.
- Do not remove files merely because they appear obsolete without approval.
- Never commit `.venv`, caches, logs, secrets or temporary test directories.
- Keep generated and binary artifacts intentional; call out material repository
  size increases.
- Do not stage, commit, push, create branches or open pull requests unless the
  user asks.
- Keep each change focused and make its purpose clear in the handoff.
- Report the exact files changed and verification commands run.

## Agent responsibilities

### General responsibilities

Every agent must:

1. Read this file before acting.
2. Inspect the relevant implementation and current git state.
3. Stay within its assigned file boundaries.
4. Preserve the current analytical meaning unless authorised to change it.
5. Verify its work in proportion to risk.
6. Return:
   - files changed;
   - work completed;
   - checks and commands run;
   - results;
   - unresolved defects;
   - risks and recommended follow-up.

Agents must not stage or commit their own changes unless explicitly instructed.
The primary agent is responsible for integration and final verification.

### Testing Agent

- Owns `tests/`, `pytest.ini`, development test dependencies and CI test
  configuration when assigned.
- May report production defects but must not fix production code unless its scope
  is expanded.
- Must emphasise edge cases, regression protection and cross-platform behaviour.

### Dashboard Polish Agent

- Owns presentation changes in `app.py` and explicitly assigned dashboard
  assets.
- Must not change scoring, transformations or source data.
- Must complete browser-based visual and interaction checks.

### Documentation Agent

- Owns `README.md`, `project_status.md` and assigned files under `docs/`.
- Must verify technical claims against code and authoritative sources.
- Must not change implementation to make documentation true.

### Code Cleanup Agent

- Owns assigned production modules under `src/` and explicitly assigned runtime
  configuration.
- Must preserve output values and schemas unless behavioural changes are
  approved.
- Must compare before-and-after outputs and report any difference.
- Must request corresponding tests from the Testing Agent when behaviour changes.

### Primary agent

- Defines non-overlapping scopes before parallel work.
- Resolves conflicts and reviews all agent outputs.
- Runs the complete pipeline, automated tests and application verification after
  integration.
- Updates the user with combined results and remaining risks.
